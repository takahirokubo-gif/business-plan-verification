# 事業計画検証ソリューション Lv1 機能要件定義（v5）

正本：JSOL銀行業界作業シート「銀行事業案②：事業計画検証ソリューションにおけるLv1開発要件」
本書はその思想を、実装可能な機能要求・機能要件に展開したもの。

---

## 1. 思想・方針（正本より）

- **GOAL**：案件情報入手後すぐに、決裁者と論点（ストレス仮説・説明ストーリー）の擦り合わせ＝**審査相談**ができる状態を作る
- **計算をさせない**：Lv1は再計算エンジンを持たない。数値は「概ねモデル内にある数値を拾ってくる」（値の拾い上げ・転記）。KPI構造は数式の構文解析のみ（再計算なし）。インパクトはLLMによる定性推定（簡易的でも具体的な数値を必須とする）
- **AIの役割は抽出・マッピング・解釈・仮説生成に限定**し、すべてのAI出力は人のレビューを経て「確定」される。AI推定値と抽出事実はUI上で常に区別する
- 本プロダクトの第一段階は**事実の整理**である。案件登録とは「案件の基礎情報と財務データという事実を、根拠付きで確定させる行為」を指す

## 2. 全体構造：4ステージとデータの状態遷移

すべてのデータ（数値・KPI・シナリオ）は「AI提案 → 人がレビュー → **確定**」の状態を持ち、確定データのみが次ステージの入力になる。

```
ステージ1：案件登録（事実の整理・数値確定）
   ↓ 確定した数値
ステージ2：KPI構造の提案・確定（構造の解釈）
   ↓ 確定したKPI
ステージ3：シナリオ検討・確定（仮説）
   ↓ 確定したシナリオ
ステージ4：Excelエクスポート（審査相談用簡易フォーマット）
```

**巻き戻しルール（決定：警告型）**：上流の確定を解除・変更しても下流は無効化しない。下流の各要素に「上流が変更されています（EBITDA FY26：1,620→1,580）」の警告バッジを表示し、再確認を促すに留める。審査相談の反復（差戻し→再検討）の中で上流修正は日常的に起きるため、作業を壊さないことを優先する。

## 3. ステータス体系（2系統）

**作業ステータス**（ステージ進捗、自動遷移）：
資料解析中 → 数値確定中 → KPI確定中 → シナリオ検討中 → 出力済

**案件検討ステータス**（融資判断の検討状態、人が設定。審査相談メモの結論と連動）：
- 検討中（初期値）
- 再検討中（審査相談で指摘を受け持ち帰り）
- 推進（稟議手続きへ進める判断）
- 保留
- 見送り

案件一覧では両ステータスを列で表示し、案件検討ステータスでのフィルタを可能にする。

## 4. 画面構成（ハブ構造）

初回登録のみウィザード、以降は案件詳細ハブに集約。ステージ2・3では右側にチャットパネルが常駐。

| # | 画面 | 対応ステージ |
|---|------|------|
| S1 | 案件一覧 | － |
| S2 | 案件登録（基本情報＋資料アップロード） | 1 |
| S3 | 案件詳細ハブ：数値確定タブ（前提情報／財務数値） | 1 |
| S4 | 案件詳細ハブ：KPI構造タブ＋チャットパネル | 2 |
| S5 | 案件詳細ハブ：シナリオタブ＋チャットパネル | 3 |
| S6 | 案件詳細ハブ：エクスポートタブ | 4 |
| S7 | 案件詳細ハブ：審査相談メモタブ | 横断 |

## 5. 機能要件

### ステージ1：案件登録（事実の整理）

**5-1. 案件基本情報の登録**
- 項目：案件名／案件種別（LBO・MBO・事業承継・リファイナンス）／借入人（SPC）／対象会社／対象会社業種／スポンサー／クローズ予定日／審査相談予定日／担当者／資金調達構造（EV・シニアローン総額・本行取組額・エクイティ）／案件概要（フリーテキスト）
- 初期レバレッジ・LTVは入力値から自動算出して表示（これは四則演算であり「再計算エンジン」には該当しない）

**5-2. 資料アップロード**
- スロット式：財務モデル（Excel：スポンサーケース／ベースケース）、DDレポート（PDF：事業／財務／法務／税務。税務のみ任意）
- ファイルごとの解析ステータス表示

**5-3. AI抽出（正本TODO 1・2に対応）**
- 定性情報：事業要約・主要リスクをDDレポート群から抽出・要約生成
- 過去実績：財務DDまたはモデルの実績シートから直近3期のBS/PL/CFを特定・転記
- 予測数値：モデル内のシナリオシート（Sponsor Case／Base Case）と対象期間列を自動特定し、PL（売上・営業利益・経常利益・EBITDA・各利益率）／BS（買収金額→のれん・純資産・現預金）／CF（FCF）を、表記揺れを吸収（名寄せ）して転記。**再計算は行わない**

**5-4. レビューと確定（Human-in-the-loop）**
- すべての抽出項目に**根拠3点セット**を表示：
  1. 参照ファイル（例：財務DDレポート）
  2. 箇所（シート・セル番地／ページ番号、原文抜粋つき)
  3. 論理（例：「モデル上の"Net Sales"を売上高にマッピング。EBITDA算定の前段項目であることをシート構造から判断」）
- 項目ごとのアクション：そのまま確定／修正して確定／保留
- 財務モデルとDDレポートで値が食い違う項目は不整合警告を表示し、どちらを採るか人が選択
- 必須項目がすべて確定するとステージ2へ進行可能

### ステージ2：KPI構造の提案・確定

**5-5. KPI構造の自動生成（正本TODO 3に対応）**
- 財務モデルの数式を構文解析し（再計算なし）、変数の連動関係をツリー化（例：売上高＝稼働人数×月間平均稼働時間×派遣単価）
- 事業DDの定性情報と照合し、重要KPI（リスクドライバー）に★を付与して提案
- 各ノードに出典種別を表示：モデル数式由来／DD由来／手動追加（モデル外KPIは「モデル外・定性」バッジ）

**5-6. チャットによる修正（差分プレビュー・適用方式）**
- 右側チャットパネルで自然言語指示（例：「大口派遣先への売上依存度をKPIに追加」「★を稼働率から採用CPAに変更」）
- AIは変更を**差分プレビュー**（追加＝緑／変更＝黄／削除＝赤）で提示し、人が「適用」を押した時のみ反映。チャット出力の直接反映はしない
- ノードのドラッグ編集など細密なGUI編集は**スコープ外**（明示）
- KPI構造の確定操作でステージ3へ

### ステージ3：シナリオ検討・確定

**5-7. シナリオカードの標準型**（AI起案・人起案とも同一構造）
1. シナリオ名・発生要因
2. 影響を受けるKPI（確定KPIツリーから選択・複数可）
3. KPIの変化幅と根拠（例：稼働率-15%。根拠：過去の経済ショック時の業界平均下落幅〔外部データ〕）
4. 返済能力への影響：DSCR・レバレッジ・現預金等への影響を**具体的数値を含むAI推定**で記述（「Year 2でDSCRが0.9倍前後まで低下と推定」）。**「AI推定・モデル再計算なし」バッジ必須**。判定ラベル（問題なし／問題あり等）は**出さない**（決定）。推定値と含意の記述まで
5. 保全策・構造、およびスポンサー等への確認事項（Q&A候補）

**5-8. シナリオの生成・追加・ブラッシュアップ**
- AI推奨シナリオ：3類型（トップライン／コスト／イベント）をベースに生成、「AI推奨」バッジ
- 人の仮説シナリオ：「＋ 自分のシナリオを追加」。一言（例：「大口派遣先が撤退したら」）から標準型に展開
- カード単位のチャットブラッシュアップ（例：「-18%のケースも見たい」「保全策にエスクローを追加」）→ 差分プレビュー → 適用
- カードごとに採用トグル。採用＝確定シナリオ

### ステージ4：エクスポート（正本TODO 5に対応）

**5-9. Excel出力**
- 確定データのみ（案件基礎情報＋確定財務数値＋確定KPI構造＋確定シナリオ）を、行内標準フォーマットに近いExcelテンプレートの指定セルに値転記して .xlsx を即時生成
- AI推定値には注記を付す（「※インパクト数値はAIによる推定であり、財務モデルの再計算値ではありません」）
- 保留項目が残る場合は「保留N項目を除いて出力します」の確認を挟む
- PDF出力は補助として提供

### 横断機能

**5-10. 審査相談メモ**
- 対面の審査相談（擦り合わせ会議）の結果を関係者が簡易に記録：相談日／出席者／結論（続行・再検討・推進・保留・見送り）／指摘事項（各シナリオ・KPI・数値に紐付け可）／自由メモ
- 結論の記録に連動して案件検討ステータスを更新（確認ダイアログを挟む）
- 指摘事項は紐付け先のシナリオカード等に「前回審査相談での指摘」として表示され、再検討ループの起点になる

**5-11. トレーサビリティと履歴**
- すべてのAI出力に出典（またはAI推定である旨）を保持
- 確定・修正・確定解除の操作履歴（誰が・いつ・何を）を案件単位で記録

**5-12. モックモード**
- `EXTRACTOR_MODE=mock`：API不要。解析実行後、疑似ディレイを挟んでオートスタッフ中部LBO案件の一気通貫デモデータ（S1〜S7が矛盾なく繋がる状態）を展開
- 実API切替は環境変数のみで可能な構成

## 6. スコープ外（Lv1で作らないもの）

- 再計算エンジン（ストレス値を財務モデルに当てた再計算）→ Lv2候補
- KPIツリーの細密なGUI編集（ドラッグ、数式の直接定義）
- 承認ワークフロー（申請・承認・差戻しの電子化）
- 認証・権限管理（デモは操作ユーザー切替ドロップダウンで代替）
- 外部市場データのライブ連携（Lv1は根拠テキストとして保持するに留める）

## 7. 想定技術構成

React + Vite + TypeScript + Tailwind（フロント）／FastAPI + SQLite（バック）／Anthropic API（抽出・解釈・チャット、mock切替可）。コベナンツ・モニタリングとデザイントークン共通（#1A4F8B、Noto Sans JP、Inter tabular、4px角丸、1pxボーダー、Tint 10%バッジ、サイドバー固定）。
