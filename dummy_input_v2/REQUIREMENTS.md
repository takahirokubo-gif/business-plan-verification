# ダミーデータ要件の幅出し（実AI精度テスト用・理想的インプット）

目的：**生成AIを実際に動かした際に（EXTRACTOR_MODE=anthropic）、正しく動くか・精度は十分かをテストする**。
そのための「理想的なインプット」＝ **必要な情報が揃っている ∧ 実際に想定されるインプットに十分近い** データを定義する。

参照した実サンプル（ユーザー提供・Google Drive）：

- DD資料：事業DD報告書サンプル（コンサル系スライド様式）／財務DD報告書サンプル①（税理士事務所様式）・②（FAS様式）
- 財務モデル：Company A LBOモデル各種（Cover/inp/calc/debt構成・百万円単位・A/P表記・ケース表）／財務モデルのサンプル（算出根拠列つき）／財務三表シミュレーションモデル（月次・#REF!エラー含む）

## 1. 目的からの逆算：このデータで何を測るか

| # | テスト対象（AIの仕事） | 対応する画面 | 測り方 |
|---|---|---|---|

| T3 | 数値の拾い上げ・転記（実績3期PL/BS/CF、計画5年×2ケース、単位換算、表記揺れの名寄せ） | S3 | 24項目の値の完全一致（丸め規約込み） |

## 2. 要件の幅出し

### A. テストデータとして成立するための要件（答え合わせ可能性）

| 要件 | 採否 | 実装 |
|---|---|---|
| 一意な正解値を持つ（すべての数値が正本spec.pyから決定論的に導出される） | ○ | spec.py＝Single Source of Truth。generate系はすべてspecから生成 |
| 根拠位置が固定・検証可能（キーファクトのページ・セル番地を構成的に保証） | ○ | PDFは1要素=1ページ生成・validate.pyがpypdfで照合。Excelはxl_layout.pyで座標管理 |
| 期待アウトプットが機械採点できる形式である | ○ | expected_output.json＋run_ai_test.py（表記揺れ・年度キーの正規化を吸収） |
| 生成タスク（KPIツリー・シナリオ）は一意解がないことを明示し、ルーブリック採点＋人手レビューに分離 | ○ | expected_output.jsonのkpi_tree/scenariosセクション |
| 既存デモデータと独立（数値・社名・業種・ファイル名の衝突なし） | ○ | 別会社・別業種・別単位系・別ディレクトリ（dummy_input_v2/） |
| 再現性（再生成しても同じ値）・変更時の整合チェック | ○ | generate_all.py → validate.py |

### B. 「必要な情報が揃っている」ための要件（充足性）

| 要件 | 採否 | 実装 |
|---|---|---|
| 抽出24項目すべての値が資料内に存在する | ○ | モデル（PL/BS/CF/inp）＋DD4種に配置。validate.pyで存在検証 |
| 案件基本情報（S2フィールド）がすべて資料から読み取れる | ○ | モデルCover・inpシート＋事業DD p.4（案件の背景とスキーム） |
| 必須スロット6ファイルが揃う（モデル2ケース＋DD事業/財務/法務/税務） | ○ | 6ファイル生成 |
| 計画値の「値」がExcelファイル内に存在する（数式だけでなく計算結果も） | ○ | LibreOffice再計算でキャッシュ値を付与（Excel保存済みファイルと同状態） |
| KPIツリーが数式から一意に導ける（ドライバー→PLの参照チェーン） | ○ | inpシートのドライバー→PLのクロスシート数式（会員数は漸化式） |
| シナリオ3類型それぞれの材料（定量的事実）がDDにある | ○ | トップライン=リピート率76%・卸集中45%/21%／コスト=OEM依存58%・原価率／イベント=COC条項・行政指導歴・商標 |
| 単位が資料内に明記されている（単位注記の欠落で桁を誤らせない） | ○ | モデル各シート「（単位：百万円）」・DD表「単位：千円」 |
| ケース（Sponsor/Base）の判別情報がファイル内にある | ○ | Cover・各シートのケースラベル・ファイル名 |
| 不整合検知のための「答えのある不整合」が存在する | ○ | のれん：モデル6,100 vs 財務DD 5,950（差異の理由も資料に記載） |
| ハルシネーション検査のための「資料に無い項目」が定義されている | ○ | 本行取組額・担当者・審査相談予定日（行内情報） |

### C. 「実際に想定されるインプットに近い」ための要件（実サンプル準拠）

| 要件 | 採否 | 実装（実サンプルの根拠） |
|---|---|---|
| LBOモデルの実務様式：Cover/inp/debt等のシート構成・小文字シート名 | ○ | Company A LBOモデル準拠（Cover, inp, PL, BS, CF, debt, memo） |
| 金額単位は百万円（million）表記 | ○ | Company Aモデル「単位, million」準拠。既存デモ（千円）と意図的に変える |
| 年度ヘッダーにA（実績）/P（計画）表記・「2026/3期」形式 | ○ | Company Aモデルの「17/12/31・A/P」表記を和式に翻案 |
| 直接入力は青字・数式は黒字の実務慣行 | ○ | 「財務モデルのサンプル」の注記に準拠（青字入力セル） |
| 実績=値貼付・計画=実数式（Excel保存によるキャッシュ値つき） | ○ | 実務のモデルと同じ状態。KPIパースと値拾いの両方が可能 |
| DD報告書の定型様式：表紙・免責事項・目次・調査基準日・「Confidential」 | ○ | 財務DDサンプル①②の冒頭構成を踏襲 |
| 財務DDの章立て：会社概要・株主の状況・会計方針・3期比較・BS再評価（時価純資産）・正常収益力（QoE）・のれん試算・総括 | ○ | 財務DDサンプル②（株主の状況・経営指標推移・SWOT）＋①（再評価BS・収益性の検証）を合成 |
| 事業DDの章立て：エグゼクティブサマリー・市場環境・ビジネスモデル・業績構造・SWOT・（買収側視点の）計画評価 | ○ | 事業DDサンプル（市場動向・3ヶ年比較損益・損益構造）を買側DDに翻案 |
| 表記揺れ：モデル「売上収益/EBIT」vs DD「売上高/営業利益」等の名寄せが必要 | ○ | 意図的に混在 |
| 単位の混在：モデル=百万円・DD表=千円（換算が必要） | ○ | 財務DDサンプルの千円表記に準拠。丸め整合（ROUND(千円/1000)=百万円）は構成的に保証 |
| 数値でない周辺情報（沿革・役員・免責文）のなかに目的の数値が埋まっている | ○ | DD各章のボリューム（計78ページ） |
| 軽微なノイズ（作業メモシート） | ○ | memoシート（旧単位・PPA未確定のメモ） |
| 意匠（日本語フォント・ヘッダー/フッター・ページ番号） | ○ | Noto Sans JP埋め込み（テキスト抽出可能＝実AIの入力経路と同じ） |

### D. 現行システムの制約への適合（「理想的インプット」の暗黙条件）

現行の `AnthropicExtractor` の読み取り仕様に収まるよう設計（ここに収まらないデータは
「AIの精度」ではなく「パイプラインの制限」を測ることになるため、意図的に分離）：

| 制約 | 値 | 対応 |
|---|---|---|
| Excelダイジェストの読み取り窓 | シートあたり60行×12列 | 全シートを窓内に収める（validate.pyが検証） |
| PDFダイジェストの上限 | 40ページ | 最大の財務DDを34ページに設計 |
| PDFはテキスト抽出可能であること（pypdf） | － | フォント埋め込みTTFで生成（スキャン画像PDFは不可） |
| Excelはopenpyxlで読めること | － | .xlsx形式・外部リンク/マクロなし |
| 数式セルの値はキャッシュ値から拾う | － | LibreOffice再計算で付与（→ §4の改修も参照） |

### E. 今回スコープ外＝次段階のテストバリエーション（幅出しのみ）

「理想的インプット」で精度のベースラインを取った後、劣化条件を1つずつ加えて頑健性を測る：

1. **重ノイズ**：旧版シート（数値が微妙に違う）・大量の作業セル・隠しシート（既存dummy_inputでカバー済みの軸）
2. **単位注記の欠落・混在**（千円/百万円/億円が注記なしで混在）
3. **読み取り窓超過**：60行×12列を超える実物大モデル（実サンプルCompany Aは13年列×数百行）・100ページ超のDD
4. **欠損**：税務DDなし・ベースケースなし（任意スロット・必須欠落時の挙動）
5. **スキャンPDF**（画像のみ・OCRなし）→ 現行はテキスト抽出不可のため要件定義が必要
6. **壊れたExcel**：#REF!/#NAME?エラー、循環参照スイッチ（実サンプル財務三表モデルに実在）
7. **不整合多発**：3資料で3様の値・期ズレ（12月決算DD vs 3月決算モデル）
8. **他社流用の痕跡**：別社名が残ったテンプレ（識別ミスの誘発）
9. **和暦・全角数字・結合セル多用**などの表記異常
10. **英語資料**（外資系スポンサーのモデル・英文DD）

- 詳細な期待値は GROUND_TRUTH.md・expected_output.json

## 4. 作成過程で発見した現行実装の課題（テスト前に把握すべきこと）

1. **［修正済み］数式セルの値がLLMに渡らない**：`_excel_digest` が `data_only=False` のみで
   読んでいたため、計画年（数式セル）は数式文字列しか渡らず、値の転記が原理的に不可能だった。
   数式＋キャッシュ値の併記に修正（backend/app/extractors/anthropic_extractor.py）。
   これに対応して、本データセットのExcelはLibreOffice再計算でキャッシュ値を持たせている
   （openpyxlが書いただけのファイルにはキャッシュ値がなく、実務のExcel保存済みファイルと状態が異なるため）。
2. **［未対応・要判断］読み取り窓の制限**：シート60行×12列・PDF40ページ。実物（実サンプル）は
   これを超えるものが多く、実運用前に分割読み込み・シート選択などの設計が必要（§E-3）。
3. **［留意］mockモードとの排他**：本データセットはmockのフィクスチャを持たないため、
   `EXTRACTOR_MODE=mock` でアップロードすると「デモモードでは〜」で弾かれる（仕様どおり）。
